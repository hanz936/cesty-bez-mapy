## ADR-001: Use Anonymous Authentication Instead of `anon` Role

**Status:** ✅ Accepted

**Context:**
We need to support guest checkout for custom itinerary requests to minimize cart abandonment (industry standard shows 26% abandonment when registration is required).

**Decision:**
Use Supabase Anonymous Sign-In (`supabase.auth.signInAnonymously()`) instead of allowing unauthenticated access with the `anon` role.

**Rationale:**

### Why Anonymous Auth vs `anon` Role?

| Aspect | `anon` Role | Anonymous Auth |
|--------|-------------|----------------|
| **JWT Token** | ❌ No token | ✅ Has JWT token |
| **User ID** | ❌ No `auth.uid()` | ✅ Has `auth.uid()` |
| **RLS Policies** | ❌ Can't use `auth.uid()` | ✅ Can use `auth.uid()` |
| **Role in JWT** | `anon` | `authenticated` |
| **Session** | ❌ No session | ✅ Has session |
| **Upgradeability** | ❌ Can't upgrade | ✅ Can upgrade via `updateUser()` |

**Key Quote from Supabase Docs:**
> "An anonymous user assumes the **authenticated role** just like a permanent user, and JWTs for these users will have an `is_anonymous` claim."

**Consequences:**

✅ **Positive:**
- Guest checkout works seamlessly
- RLS policies can use `auth.uid()` pattern
- Anonymous users can be upgraded to permanent users
- Session persists across page navigation
- CAPTCHA protection can be enforced

⚠️ **Negative:**
- Requires CAPTCHA to prevent abuse (mandatory)
- Rate limiting needed (30/hour default)
- More anonymous user records in `auth.users` table

**Alternatives Considered:**

1. ❌ **Email-based access without auth:** Too insecure, can't use RLS
2. ❌ **`anon` role with public access:** Can't track individual users, no session
3. ✅ **Anonymous authentication:** Chosen for security + UX balance

**References:**
- [Supabase Anonymous Sign-Ins](https://supabase.com/docs/guides/auth/auth-anonymous)

## ADR-002: Use `auth_user_id` Column for RLS Instead of `customer_id`

**Status:** ✅ Accepted (CRITICAL FIX)

**Context:**
Initial plan used `customer_id = auth.uid()` in RLS policies, but this created a schema mismatch that broke guest checkout.

**Problem Identified:**

```sql
-- ❌ BROKEN: Original approach
CREATE TABLE custom_itinerary_requests (
  customer_id uuid REFERENCES customers(id)  -- customers.id is own UUID!
);

CREATE POLICY "broken"
  USING (customer_id = auth.uid());  -- auth.uid() ≠ customers.id !!!
```

**Decision:**
Add a separate `auth_user_id` column that directly references `auth.users.id` and use it for RLS policies.

**Rationale:**

### Schema Mismatch Explanation

```
customers.id (own generated UUID)
  ↓
  Generated via: DEFAULT gen_random_uuid()
  Example: 12345678-1234-1234-1234-123456789012

auth.users.id (Supabase Auth UUID)
  ↓
  Generated by: Supabase Auth system
  Example: 87654321-4321-4321-4321-210987654321

❌ These are DIFFERENT UUIDs!
```

### Dual-Column Pattern

| Column | References | Purpose | Value for Anonymous | Value for Permanent |
|--------|------------|---------|---------------------|---------------------|
| `auth_user_id` | `auth.users.id` | RLS policies | UUID from auth | UUID from auth |
| `customer_id` | `customers.id` | Business logic | NULL | UUID from customers |

**Code Pattern:**

```sql
-- ✅ CORRECT: Separate columns for different purposes
ALTER TABLE custom_itinerary_requests
  ADD COLUMN auth_user_id uuid REFERENCES auth.users(id);

CREATE POLICY "correct"
  USING (auth_user_id = auth.uid());  -- ✅ Both are auth.users.id
```

**Why This Works:**

```
Anonymous User Flow:
  auth.users: { id: abc-123, is_anonymous: true }
  customers: (NO RECORD)
  custom_requests: { auth_user_id: abc-123, customer_id: NULL }
  RLS: auth_user_id = auth.uid() → abc-123 = abc-123 ✅

Permanent User Flow:
  auth.users: { id: xyz-789, is_anonymous: false }
  customers: { id: xyz-789 } (created by trigger)
  custom_requests: { auth_user_id: xyz-789, customer_id: xyz-789 }
  RLS: auth_user_id = auth.uid() → xyz-789 = xyz-789 ✅
```

**Consequences:**

✅ **Positive:**
- RLS policies work for both anonymous and permanent users
- Clear separation of concerns (auth vs business logic)
- Follows Supabase best practices

⚠️ **Negative:**
- Additional column in table (minor storage overhead)
- Need to maintain both columns during lifecycle

**Alternatives Considered:**

1. ❌ **Use only `customer_id`:** Doesn't work for anonymous users (NULL)
2. ❌ **Make `customers.id = auth.users.id` via constraint:** Can't because `gen_random_uuid()` already set
3. ✅ **Dual-column pattern:** Chosen for clarity and correctness

**References:**
- [Supabase User Management](https://supabase.com/docs/guides/auth/managing-user-data)
- [Row Level Security](https://supabase.com/docs/guides/database/postgres/row-level-security)

## ADR-003: Automate Customer Lifecycle with Database Triggers

**Status:** ✅ Accepted

**Context:**
Need to synchronize `customers` table with `auth.users` lifecycle events (sign-up, anonymous→permanent upgrade).

**Decision:**
Use PostgreSQL triggers with `SECURITY DEFINER` functions to automatically create and link customer records.

**Rationale:**

### Manual vs Automated

| Approach | Pros | Cons |
|----------|------|------|
| **Manual** | Simple, explicit | Error-prone, inconsistent, forgotten steps |
| **Automated (Triggers)** | Consistent, zero-maintenance | More complex setup, harder to debug |

**We chose automated because:**
1. Prevents human error (forgetting to create customer)
2. Ensures data consistency
3. Handles edge cases (anonymous upgrade)
4. Zero code changes in application

**Trigger Architecture:**

```
Event: auth.users INSERT (permanent user)
  ↓
  Trigger: on_auth_user_created
  ↓
  Function: handle_new_permanent_user() [SECURITY DEFINER]
  ↓
  Action: INSERT into customers

Event: auth.users UPDATE (email added)
  ↓
  Trigger: on_auth_user_email_set
  ↓
  Function: handle_user_email_update() [SECURITY DEFINER]
  ↓
  Action: INSERT into customers

Event: customers INSERT
  ↓
  Trigger: on_customer_created
  ↓
  Function: link_requests_to_customer() [SECURITY DEFINER]
  ↓
  Action: UPDATE custom_itinerary_requests SET customer_id
```

**SECURITY DEFINER Requirement:**

From Supabase community:
> "When a trigger operated by the `supabase_auth_admin` interacts outside the auth schema, it causes a permission error. A security definer function retains the privileges of the database user that created it."

```sql
-- ✅ MUST use SECURITY DEFINER
CREATE FUNCTION handle_new_permanent_user()
RETURNS trigger
LANGUAGE plpgsql
SECURITY DEFINER  -- ← CRITICAL!
SET search_path = public
AS $$ ... $$;
```

**Consequences:**

✅ **Positive:**
- Zero-maintenance customer creation
- Handles all edge cases automatically
- Data consistency guaranteed
- No application code changes needed

⚠️ **Negative:**
- More complex database schema
- Harder to debug (need to check logs)
- If trigger fails, signup could be blocked

**Mitigation:**
- Extensive testing of all lifecycle scenarios
- Monitoring of Supabase logs
- `ON CONFLICT DO NOTHING` to prevent failures

**Alternatives Considered:**

1. ❌ **Application-side logic:** Error-prone, duplicated across frontend/backend
2. ❌ **Edge Functions:** Extra latency, harder to maintain
3. ✅ **Database triggers:** Chosen for reliability and performance

**References:**
- [Supabase Triggers](https://supabase.com/docs/guides/database/postgres/triggers)
- [Medium: Supabase Triggers Guide](https://medium.com/@ctrlaltmonique/how-to-insert-usernames-into-profiles-table-using-supabase-triggers-ef14d98747da)

## ADR-004: Use HTML + window.print() for PDF Generation

**Status:** ✅ Accepted

**Context:**
Need to generate PDFs of custom itinerary previews for users to download.

**Decision:**
Use native browser printing (`window.print()`) with `@media print` CSS instead of libraries like jsPDF or Puppeteer.

**Rationale:**

### Comparison of Approaches

| Approach | Setup | Czech Diacritics | Control | File Size | Dependencies |
|----------|-------|------------------|---------|-----------|--------------|
| **jsPDF** | Complex | ⚠️ Requires fonts | Medium | Large | +500KB |
| **Puppeteer** | Backend needed | ✅ Perfect | High | Large | Backend service |
| **window.print()** | Minimal | ✅ Perfect | High | Native | None |

**Why window.print() Wins:**

1. ✅ **Zero dependencies** (no bundle bloat)
2. ✅ **Perfect Czech support** (browser handles diacritics)
3. ✅ **Native PDF dialog** (users can choose settings)
4. ✅ **Full CSS control** via `@media print`
5. ✅ **A4 page sizing** via `@page` rules
6. ✅ **Best practice 2026** (no need for server-side rendering)

**Implementation:**

```css
@media print {
  /* Hide UI elements */
  .no-print { display: none !important; }

  /* Page setup */
  @page {
    margin: 2cm;
    size: A4;
  }

  /* Typography for print */
  body {
    font-family: Georgia, 'Times New Roman', serif;
    font-size: 12pt;
    line-height: 1.6;
    color: #000;
  }

  /* Page break control */
  h2 { page-break-after: avoid; }
  section { page-break-inside: avoid; }
}
```

**User Experience:**

1. User clicks "Stáhnout PDF"
2. `window.print()` opens native dialog
3. User can:
   - Preview PDF
   - Adjust margins
   - Choose pages
   - Save as PDF or print

**Consequences:**

✅ **Positive:**
- No bundle size increase
- Works offline
- Perfect Czech diacritics
- User has full control (margins, orientation, etc.)
- No backend infrastructure needed

⚠️ **Negative:**
- User must manually save (not automatic download)
- Styling requires `@media print` knowledge
- Can't enforce exact pixel-perfect layout

**Alternatives Considered:**

1. ❌ **jsPDF:** Bundle size too large, Czech diacritics problematic
2. ❌ **Puppeteer:** Requires backend service, overkill
3. ❌ **react-pdf:** Complex setup, large dependencies
4. ✅ **window.print():** Chosen for simplicity and zero-dependency approach

**References:**
- [MDN: Printing Guide](https://developer.mozilla.org/en-US/docs/Web/CSS/@media/print)
- [@page CSS Spec](https://developer.mozilla.org/en-US/docs/Web/CSS/@page)

## ADR-005: Use Cloudflare Turnstile for CAPTCHA

**Status:** ✅ Accepted

**Context:**
Anonymous sign-ins are vulnerable to abuse (bots creating unlimited users). Need CAPTCHA protection.

**Decision:**
Use Cloudflare Turnstile (invisible mode) instead of Google reCAPTCHA.

**Rationale:**

### Comparison

| Feature | reCAPTCHA v3 | reCAPTCHA v2 | Cloudflare Turnstile |
|---------|--------------|--------------|----------------------|
| **Privacy** | ⚠️ Google tracking | ⚠️ Google tracking | ✅ Privacy-focused |
| **Cost** | Free (1M/month) | Free | Free (unlimited) |
| **UX (Invisible)** | ✅ Yes | ❌ No (checkbox) | ✅ Yes |
| **False Positives** | ⚠️ High | Low | Low |
| **EU GDPR** | ⚠️ Problematic | ⚠️ Problematic | ✅ Compliant |
| **2026 Trend** | Declining | Legacy | ✅ Growing |

**Why Turnstile Wins:**

1. ✅ **Privacy-first** (no Google tracking)
2. ✅ **GDPR compliant** (important for Czech/EU users)
3. ✅ **Free unlimited** (no usage caps)
4. ✅ **Invisible mode** (best UX)
5. ✅ **Lower false positive rate**
6. ✅ **Official Supabase integration**

**Implementation:**

```jsx
import { Turnstile } from '@marsidev/react-turnstile';

<Turnstile
  siteKey={import.meta.env.VITE_TURNSTILE_SITE_KEY}
  onVerify={(token) => setCaptchaToken(token)}
  options={{
    theme: 'light',
    size: 'invisible',  // ← Best UX
  }}
/>
```

**Security Enforcement:**

```js
// Frontend: Pass token
await supabase.auth.signInAnonymously({
  options: { captchaToken }
});

// Backend: Supabase validates token automatically
// No additional server-side code needed! ✅
```

**Consequences:**

✅ **Positive:**
- Prevents bot abuse
- Better UX than reCAPTCHA
- GDPR compliant (important for EU)
- Zero cost
- Official Supabase support

⚠️ **Negative:**
- Another third-party dependency (Cloudflare)
- Requires account setup
- Can fail if Cloudflare is down

**Mitigation:**
- Rate limiting as fallback (30/hour per IP)
- Monitor anonymous user creation rate
- Alert on unusual spikes

**Alternatives Considered:**

1. ❌ **No CAPTCHA:** Vulnerable to abuse
2. ❌ **reCAPTCHA v3:** Google tracking, GDPR concerns
3. ❌ **reCAPTCHA v2:** Bad UX (checkbox)
4. ❌ **hCaptcha:** Similar to reCAPTCHA, less privacy-focused
5. ✅ **Cloudflare Turnstile:** Chosen for privacy + UX

**References:**
- [Cloudflare Turnstile Docs](https://developers.cloudflare.com/turnstile/)
- [Supabase CAPTCHA Guide](https://supabase.com/docs/guides/auth/auth-captcha)

## ADR-006: Simplify Navigation (No useTransition)

**Status:** ✅ Accepted

**Context:**
Initial plan used React 19's `useTransition` for navigation after form submission.

**Decision:**
Use simple `navigate()` after async operation completes instead of wrapping in `startTransition`.

**Rationale:**

### useTransition Purpose

From React docs:
> "`useTransition` allows you to mark state updates as **non-urgent** so React can interrupt them to handle more urgent updates (like user input)."

**When to Use useTransition:**

✅ **Good:**
- Switching between tabs (non-urgent UI update)
- Filtering large lists (can be interrupted)
- Updating complex UI based on props

❌ **Bad:**
- Navigation after async API call (not interruptible)
- Simple loading states
- Form submissions that must complete

**Our Case: Form Submission + Navigation**

```jsx
// ❌ WRONG: Over-complicated
const [isPending, startTransition] = useTransition();

startTransition(async () => {
  await insertRequest();  // External API - can't interrupt!
  navigate('/preview');
});

// ✅ CORRECT: Simple and clear
const [loading, setLoading] = useState(false);

setLoading(true);
await insertRequest();
navigate('/preview');
setLoading(false);
```

**Why Simple Approach is Better:**

1. ✅ More readable (clear intent)
2. ✅ Proper loading state management
3. ✅ Works with async/await naturally
4. ✅ Can't be interrupted (correct for this use case)
5. ✅ No unnecessary re-renders

**Consequences:**

✅ **Positive:**
- Simpler code
- Easier to understand
- Proper loading state
- Better error handling

⚠️ **Negative:**
- None (useTransition was wrong choice here)

**Alternatives Considered:**

1. ❌ **useTransition:** Wrong pattern for external async operations
2. ✅ **useState + async/await:** Chosen for clarity and correctness

**References:**
- [React useTransition](https://react.dev/reference/react/useTransition)
- [React Router Transitions](https://reactrouter.com/en/main/hooks/use-transition)

## ADR-007: Post-Purchase Account Creation (Optional)

**Status:** ✅ Accepted

**Context:**
Anonymous users complete checkout but don't have accounts. Need way to convert them to permanent users.

**Decision:**
Show optional account creation prompt after successful purchase using `supabase.auth.updateUser()`.

**Rationale:**

### E-Commerce Best Practices 2026

From industry research:
- **26% cart abandonment** when registration is required upfront
- **60% conversion rate** for post-purchase account creation
- **One-click signup** with pre-filled email is key

**Our Approach:**

```jsx
<PostPurchaseAccountPrompt orderData={...}>
  {/* Pre-filled email (read-only) */}
  <Input value={orderData.customer_email} disabled />

  {/* Only ask for password */}
  <Input type="password" placeholder="Create password" />

  {/* Optional skip */}
  <Button>Vytvořit účet</Button>
  <Link>Ne, děkuji</Link>
</PostPurchaseAccountPrompt>
```

**Why This Works:**

1. ✅ **Zero friction** during checkout (guest allowed)
2. ✅ **Reduced effort** (only password, email pre-filled)
3. ✅ **Preserves data** (same `auth.users.id`, triggers link everything)
4. ✅ **Optional** (respects user choice)
5. ✅ **Timing** (after successful payment = high trust moment)

**Technical Implementation:**

```js
// Convert anonymous → permanent
await supabase.auth.updateUser({
  email: orderData.customer_email,
  password: password,
});

// Triggers automatically:
// 1. handle_user_email_update() → Create customer
// 2. link_requests_to_customer() → Link requests
// 3. link_orders_to_customer() → Link orders

// auth.users.id stays the same! ✅
```

**Consequences:**

✅ **Positive:**
- Higher conversion than forced registration
- Better customer retention
- Seamless data linking
- Respects user autonomy

⚠️ **Negative:**
- Some users remain anonymous (can't reach later)
- Email verification required (if enabled in Supabase)

**Alternatives Considered:**

1. ❌ **Force registration upfront:** 26% abandonment rate
2. ❌ **Magic link only:** Requires email access immediately
3. ❌ **No account creation:** Can't track returning customers
4. ✅ **Optional post-purchase:** Chosen for optimal conversion

**References:**
- [E-Commerce Checkout Optimization 2026](https://baymard.com/checkout-usability)
- [Supabase updateUser](https://supabase.com/docs/reference/javascript/auth-updateuser)
